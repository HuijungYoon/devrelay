This page describes how the hosted Redmine DevRelay MCP service handles data. It is provided for OpenAI Plugins Directory listing and user transparency. It is not legal advice; review before relying on it commercially.

1. What the service does

Redmine DevRelay connects ChatGPT / Codex to a Redmine instance you choose via the Model Context Protocol (MCP). Users supply their own Redmine base URL and API key (BYOK). The service calls that Redmine’s REST API to read or (with explicit confirm) write issues.

2. Data you provide

  • Redmine URL and API key — sent as request headers (X-Redmine-Url, X-Redmine-Api-Key) when you connect.
  • Tool inputs — issue ids, subjects, comments, filters, and similar fields you ask the assistant to send.
  • Redmine responses — issue/project/user fields returned by your Redmine and passed back to the AI host.

3. How we use data

  • Credentials and tool inputs are used only to fulfill the MCP request (proxy to your Redmine).
  • Production hosted MCP keeps HTTP_ALLOW_ENV_FALLBACK=0: no shared process-wide Redmine key for end users.
  • Session state may hold an in-memory Redmine client for the MCP session lifetime (idle TTL / max sessions apply). Sessions are not a long-term credential vault.
  • We do not sell personal data or use Redmine contents for advertising.

4. Hosting and subprocessors

The production MCP endpoint is hosted on Fly.io. Infrastructure logs (e.g. request timing, error codes, hostnames) may be retained by the host per their policies. Avoid putting secrets in issue text or logs.

5. Demo / review instance

A separate public demo Redmine may be provided solely for OpenAI review and documentation. It holds disposable seed data and is not your corporate Redmine. Demo credentials must not be reused for production systems.

6. Local / Git marketplace installs

When you run npx redmine-devrelay locally (STDIO), credentials stay in your environment and process. That path does not require the hosted MCP.

7. Retention

We do not operate a user account database for BYOK keys. In-memory sessions expire (TTL) or are closed with the MCP session. Hosting provider logs follow Fly.io retention. You control retention inside your own Redmine.

8. Security

  • Prefer HTTPS Redmine URLs.
  • Do not paste API keys into chat transcripts when avoidable.
  • Write tools use dry-run → confirm + previewToken; treat confirm as deliberate approval.

9. Children’s data

The service is intended for workplace / developer use, not for children under 13 (or the equivalent age in your jurisdiction).
